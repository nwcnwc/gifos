# Handoff — 2026-07-20 (session 2: the battery, and what it found)

Continues `docs/handoff-2026-07-20.md`. Written to be read cold.

## The headline

Two of the four browser failures in the join battery were the TEST FLEET, not
the product, and both are now fixed in the battery itself. The other two are
real and are described below. Separately, the "joining thins out past 5% packet
loss" mystery is **diagnosed**: it is not thinning, it is a permanent wedge, and
the obvious fixes are wrong — it needs a decision.

## The fleet traps (fixed)

1. **nvidia-laptop defaults to node 18, which has no global `WebSocket`.**
   `gifos-net.js` opens the relay socket with `new WebSocket`; under node 18 the
   constructor throws, the connect path catches it and reschedules forever, and
   `flood.js` / `e2e-mesh-wire.js` **hang** — no output, until the step timeout.
   That is indistinguishable from a genesis deadlock and it cost a full battery
   run. node 22 is now installed there (`nvm use 22`, default), and the battery
   refuses to start on a node without the global.
2. **The battery never started the dev stack.** The browser ladders drive
   `swarm.js` at `--base 8099 --relay 8790`; with nothing listening, every bot
   fails to load and each rung reports `seated=?/N`. All nine rungs "failed",
   including N=2. The battery now brings site+relay up itself (`RELAY_DEV=1`)
   when those ports are idle, and aborts loudly if it cannot.

Both are the same lesson the last session learned twice: **when a browser run
disagrees with the sim, suspect the fleet first.**

## The split-room question — ANSWERED, with a caveat

`drills/adversary-room.js` now CLASSIFIES every duplicate coord instead of just
reporting it. The three cases, and why only one is a bug:

| kind | meaning | verdict |
|---|---|---|
| `PARTITIONED` | a holder with NO completed links at all | accepted — the sim reproduces it (split-brain allowed, detection-only) |
| `PAIR-DARK` | both holders wired into the room, no channel to each other, no common witness | the REUNION question, not a seating bug |
| `REACHABLE` | the holders are connected, or some third participant holds live links to BOTH | a real fault — E2's yield law had a witness and the duplicate stood anyway |

The classifier tests for a **common first-hand-live witness**, because that is
what the yield law actually requires (`mesh.js` E2, ~line 934): gossip echoes
never yield.

**Result over two runs: every duplicate was `PARTITIONED`. Zero `REACHABLE`.**
So the browser case from last session was the accepted partition, not healthy
peers sharing cells. `no two MUTUALLY REACHABLE peers share a cell` now gates
separately from the distinct-coord check.

**The caveat, and it is the real finding.** In those runs, *healthy* clients had
zero completed channels — `late1`, `late2` and `final` in one run, and in the
standalone re-run two healthy late joiners never seated at all while a third
FOUNDED ITS OWN ROOM (`0/0.0`, population 2). With the relay fallback gone a
client that cannot open DataChannels is partitioned by definition, so the
accepted-but-rare condition is now **common**, exactly as the last handoff
predicted. That is the reunion decision, and it now has teeth.

The sim does NOT reproduce it: a dark seat at a row head in a small room still
admits every newcomer, before and after any of the fixes tried below. The sim's
newcomer reaches a severed greeter over the modelled relay just fine. So this
one lives in the **wire/greeter layer**, which is precisely the gap
`test/README.md` names — the sim models whether two seats CAN reach each other,
never whether they have finished negotiating. **Next step: re-run
`adversary-room.js` with `RELAY_DEBUG=1` and per-user traces, and find out why a
healthy late joiner gets an empty greeter list (it founds a room only on an
empty list — `mesh.js` GREETERS handler, R3 mint).**

## The loss wedge — diagnosed, NOT fixed, needs a decision

`net loss=0.10` at 150 seats seats 5 people, forever. It is not statistical
thinning; it is a deterministic wedge, and here is the whole mechanism:

1. A newcomer is PLACEd into a Section-1 **row head**, say `/1.0`. The PLACE is
   lost. `admit()` has already written `occ[/1.0] = thatPeer` and `noteS1`'d it.
2. The peer never seats — it stays `searching` forever. Nothing ever clears the
   entry: `s1Fill` sweeps only the head's OWN row, and the vertical healer of a
   Section-1 row head is a down-child that does not exist yet. Observed:
   `moves=0 evict=0 healAt=-99` after 400k ticks.
3. Row 1 now reads as **"seen but not live"**, so `serveFind` takes the
   RESURRECTION branch and delegates to the row below (row 2) — which is empty.
   It `continue`s. The cell scan never even reaches row 1.
4. Every cell of row 2 names `/1.0` as its designated admitter, and the hand-off
   uses **raw `occ`**, so those FINDs are handed to a peer that is not seated
   and are swallowed.

Result: the room is capped at row 0 — five seats — and 55 of 60 joiners search
forever. Reproduce with:

    printf "net loss=0.10\njoinmode burst\ninit 60\nconverge 400000\nstate\nbad\nseat 0\nseat 6\nquit\n" | /tmp/mesh --service

`seat 0` shows `/1.0=6`; `seat 6` shows `state=searching`, forever.

### Two candidate fixes, both tried, both rejected

**(a) Gate the admitter hand-off on `firstHandLive`** — mirroring the sibling
branch a few lines above, whose comment says using raw `occGet` there *was* a
bug. Seating is dramatically better:

| fabric (burst 150) | before | after |
|---|---|---|
| loss 0.10 | 5 | 150 |
| loss 0.20 | 4 | 150 |
| loss 0.30 | — | 131 |
| subnets=6 density=0.60 | 9 | 150 |

**But it wrecks the healthy path**: `burst N=100` converges in 63 SECONDS
instead of 29 ms. A legitimate admitter is often known only by gossip, so the
gate drops good hand-offs and joiners fall back onto the 60-tick retry. Correct
seating, unusable convergence. Reverted.

**(b) Expire an unconfirmed admission** — `admit()` stamps `placedAt[cell]`, and
the cell is re-offered if its occupant has never spoken first-hand within
`PLACE_TTL`. Principled, and it does not touch the healthy path — but it does
not fix the wedge either, because of step 3 above: the scan never reaches the
cell, so a cell-scan fix cannot fire. Rejected on evidence.

### What is actually owed

The block is upstream of both: a **phantom PLACE marks the row `s1seen`**, which
routes the whole row into resurrection, whose designated resurrectors are a row
that does not exist. Candidate directions, all of which touch a documented law
and so are **Nathan's call**:

- do not `noteS1` a cell on PLACE — only on confirmed occupancy;
- make the resurrection branch fall through to ordinary admission when the
  admitter row is empty, instead of `continue`;
- give a Section-1 row head with no down-child a designated healer.

This matters more than a sim curiosity: 10% loss is an ordinary phone network.

## State of the battery

dark seat, `mesh.js` harness (N=1000, 50% kill, s1row, s1all, D5), genesis

owned links include a socketless seat, and places a normally-arrived joiner

under test is unchanged — signaling to a link target with no relay socket — it
is simply exercised on every run now. Needs `DEBUG=on`, so every user in that
drill loads with it.
